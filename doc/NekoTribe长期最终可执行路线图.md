## 1. 文档定位

这份路线图不是愿景清单，而是面向当前仓库状态制定的长期执行计划。

它解决三个问题：

1. 明确未来 24 个月应该先做什么、后做什么。
2. 避免功能扩张过快，导致 V1 技术债继续累积。
3. 给后续开发、联调、测试、上线、运营提供统一节奏和阶段退出标准。

## 2. 制定路线图时采用的现实假设

### 2.1 团队与资源假设

- 按个人项目或 1 至 3 名核心开发者的节奏规划。
- 默认没有独立专职 QA、运维、设计、运营，相关工作需穿插完成。
- 如后续团队扩充到 4 至 5 人，可压缩时间，但阶段顺序不建议打乱。

### 2.2 技术假设

- 继续采用 `Nuxt 3 + Nitro + Oracle + Redis + WebSocket` 作为主技术路线。
- Oracle 作为主数据源保留，不切换数据库体系。
- Web 端优先，移动端在 Web 稳定后推进。
- 现有 `v1` 代码继续保留一段兼容期，不立即废弃。

### 2.3 当前仓库现实判断

截至 2026-04-24，项目已经具备较完整的社交平台雏形，但仍存在以下关键约束：

- 已有较完整的 `server/api/v1/**` 模块和前端页面。
- 已有 Oracle、Redis、JWT、上传、通知、群组、聊天、审核等方向的基础代码。
- 尚未建立正式的 `server/api/v2/**` 目录和完整迁移闭环。
- `user settings`、`sessions`、`statements` 仍存在本地 JSON 演示存储。
- 群组、聊天、审核等页面仍有较多模拟数据和原型态实现。
- `package.json` 尚未形成标准化测试脚本与质量闸门。

结论只有一个：

未来 6 个月的核心重点必须是“架构收口 + 数据收口 + 原型转正式”，而不是继续无序扩张新功能。

## 3. 最终目标状态

## 3.1 2028-04-30 前要达到的产品终局

到 2028-04-30，NekoTribe 应达到以下状态：

- Web 主站成为可稳定运营的社交平台。
- 核心能力完整：用户、内容、互动、通知、搜索、群组、审核、账户安全。
- 实时能力稳定：通知、在线状态、基础聊天室或群组频道能力可用。
- 国际化不止停留在界面翻译，而是形成内容、文案、运营层面的多语言支持。
- AI 功能采用“辅助增强”定位，不替代核心产品逻辑。
- 商业化只在产品稳定后进行验证，不提前透支体验。

## 3.2 2028-04-30 前要达到的工程终局

- `v2 API` 成为唯一正式对外契约。
- `v1 API` 完成下线或仅保留极少数内部兼容层。
- Oracle 成为唯一正式业务数据源，移除演示用 JSON 存储路径。
- 建立统一 DTO、统一响应结构、统一错误码、统一日志标记。
- 至少具备基础单元测试、API 集成测试、关键流程冒烟测试。
- 建立发布、回滚、监控、慢查询、错误率、版本覆盖率统计机制。

## 3.3 2028-04-30 前要达到的运营终局

- 有明确的内容治理规则、申诉流程、审核记录和操作留痕。
- 有可观察的增长漏斗、留存指标、活跃指标和性能指标。
- 有定期发布节奏、变更记录和版本公告。
- 有可控的私测、公测、正式版、增量实验流程。

## 4. 路线图总原则

### 4.1 总原则

1. 先收口架构，再扩张产品。
2. 先把核心链路做真，再把演示页面做漂亮。
3. 先让数据模型稳定，再推进推荐、AI、商业化。
4. 每个阶段必须有明确交付物和退出条件。
5. 新功能只有在不破坏主线节奏时才进入当前阶段。

### 4.2 阶段执行规则

- 每个阶段默认采用两周一个迭代。
- 每月一次阶段复盘。
- 每季度至少一次可演示版本。
- 所有重要功能必须定义“完成标准”，不能只以页面存在为完成。

### 4.3 暂缓规则

在以下事项完成前，不建议提前推进高投入新方向：

- 在 `v2 API` 最小闭环未跑通前，不启动原生移动端。
- 在演示态数据未收口前，不启动复杂推荐算法。
- 在审核和风控未形成闭环前，不扩大公开用户增长。
- 在基础性能与监控未建立前，不推进商业化实验。

## 6. 各阶段详细执行路线

## 6.1 Phase 0：基线冻结与全量盘点

**时间：** 2026-05-01 至 2026-05-31

### 阶段目标

- 冻结未来 3 个月的主线边界。
- 盘点清楚仓库中哪些功能是真实可用，哪些仍是原型/模拟。
- 为后续 `v2` 迁移建立统一任务台账。

### 关键任务

1. 盘点所有 `server/api/v1/**` 接口，标记为只读、写接口、新旧表切换、高风险接口。
2. 盘点前端页面，标记真实联调页、半联调页、纯模拟页。
3. 盘点数据库对象与文档差异，形成一份实际执行清单。
4. 统一响应格式、错误码、日志字段命名规范。
5. 建立开发规则：
   - 新功能默认不再新增 `v1` 契约。
   - 新页面默认不直接写死版本路径。
   - 演示数据页面必须标记“待后端接入”。
6. 建立项目看板：
   - 主线任务
   - 技术债任务
   - 风险任务
   - 阻塞任务

### 交付物

- API 盘点表
- 页面状态盘点表
- 数据对象盘点表
- 统一错误码和响应约定
- 阶段看板与优先级标签

### 退出条件

- 所有 `v1` 模块都被纳入清单。
- 所有模拟页面都被识别出来。
- 主线任务明确冻结，不再边迁移边改契约。

### 本阶段不做

- 不启动大规模新页面开发。
- 不启动原生移动端。
- 不启动推荐算法和 AI 实验。

## 6.2 Phase 1：API v2 基础设施与最小闭环

**时间：** 2026-06-01 至 2026-07-31

### 阶段目标

- 建立 `v2` 的正式骨架。
- 跑通“前端无版本路径调用 -> 版本决策 -> v1/v2 兼容 -> 日志可观测 -> 可回滚”的完整闭环。
- 优先迁移低风险只读接口。

### 关键任务

1. 建立 `server/api/v2/**` 基础目录。
2. 更新鉴权中间件，使其支持 `v2 auth` 放行。
3. 提供统一响应工具、统一错误工具、统一 DTO 命名规范。
4. 在前端请求层增加：
   - 接口版本决策
   - 请求适配
   - 响应适配
5. 日志增加版本标记：
   - 最终路由版本
   - 请求耗时
   - 错误类型
6. 迁移首批低风险接口：
   - `users/me`
   - `users/{id}`
   - `users/search`
   - 可选的 `system/config`
7. 建立第一批自动化冒烟测试：
   - 登录后访问用户信息
   - 前端页面读取个人资料
   - 版本切换回滚验证

### 交付物

- `server/api/v2/**` 初始结构
- 统一响应工具和适配层
- 首批 `v2` 只读接口
- 版本切流与回滚日志
- 第一批基础测试

### 退出条件

- 至少 3 个只读接口可以稳定运行在 `v2`。
- 前端至少 1 个页面以“无版本路径”方式工作。
- 可以按接口粒度切回 `v1`。
- 版本覆盖率和错误率可以被统计。

### 阶段成功指标

- `v2` 覆盖率达到 10% 以上。
- `v2` 只读接口错误率不高于 `v1`。
- 基础冒烟流程可在本地和测试环境稳定通过。

## 6.3 Phase 2：数据收口与核心域重构

**时间：** 2026-08-01 至 2026-10-31

### 阶段目标

- 把最危险、最分散、最容易留下技术债的核心域正式收口。
- 让演示态存储退出正式业务链路。
- 完成 `v2` 核心数据模型的第一轮定型。

### 关键任务

1. 重构 auth/session：
   - 会话模型统一到数据库
   - token 只存 hash / jti / session_id
   - 刷新、注销、其他设备注销统一资源化
2. 将以下模块从本地 JSON 迁移到 Oracle：
   - `account/settings`
   - `account/sessions`
   - `account/statements`
3. 重构关系模型：
   - `follow`
   - `block`
   - `mute` 可选
4. 完成内容域重构：
   - `tweets -> posts`
   - `hashtags -> tags`
   - `media -> media_assets + post_media`
5. 拆分 SQL 与 DDL：
   - 按模块落盘
   - 标记迁移批次
   - 预留回滚脚本
6. 建立数据库校验能力：
   - 新旧数据对账
   - 慢查询追踪
   - 双写差异检查
7. 完成第二批 `v2` 接口迁移：
   - auth
   - posts
   - comments
   - bookmarks
   - notifications 的低风险路径

### 交付物

- `n_auth_sessions` 等正式会话模型
- Oracle 版 settings / sessions / statements
- `posts / tags / media_assets` 正式结构
- 数据迁移脚本、校验脚本、回滚脚本
- `v2` 核心写接口

### 退出条件

- 正式环境主链路不再依赖本地 JSON 存储。
- 主要认证与内容写链路可在 `v2` 运行。
- 已建立新旧数据校验机制。
- `v2` 覆盖率达到 40% 至 60%。

### 阶段成功指标

- 数据迁移缺陷可在上线前通过校验脚本发现。
- P95 API 响应时间维持在可接受区间。
- 高风险写接口具备明确回滚策略。

## 6.4 Phase 3：核心功能补齐与内测版

**时间：** 2026-11-01 至 2027-01-31

### 阶段目标

- 把目前“有页面但不完整”或“有原型但未正式接后端”的模块做成真实可用功能。
- 发布第一个可稳定拉人内测的版本。

### 关键任务

1. 完成内容域补齐：
   - 推文编辑
   - 我的点赞
   - 收藏页
   - 账户删除
   - 在线状态
2. 将群组模块从原型页升级为真实业务：
   - 群组发现
   - 我的群组
   - 邀请
   - 成员角色
   - 群组帖子
3. 将审核模块从模拟数据升级为真实业务：
   - 举报流转
   - 审核队列
   - 处理记录
   - 申诉关联
4. 将聊天模块从演示态升级为第一版正式能力：
   - 文字频道
   - 基础消息发送
   - 在线成员状态
   - 已读与未读状态可后置
5. 搜索、通知、提及、标签页做统一联调。
6. 建立内测流程：
   - 测试账号
   - 缺陷记录
   - 每周内测回收
7. 建立最低质量闸门：
   - `lint`
   - `typecheck`
   - `build`
   - 关键 API 冒烟

### 交付物

- 内测版 Web
- 群组、审核、聊天、搜索的正式后端接入
- 关键功能质量闸门
- 缺陷管理与发布说明模板

### 退出条件

- 关键主页面不再依赖模拟数据。
- 内测用户可以完成完整闭环：
  - 注册
  - 发布
  - 互动
  - 通知
  - 群组
  - 审核申诉
- 每周版本具备稳定回归流程。

### 阶段成功指标

- 形成 30 至 50 名内测用户的稳定反馈循环。
- 关键阻断性缺陷在一个迭代内完成闭环。
- 主路径页面首屏与 API 体验达到可公开测试水平。

## 6.5 Phase 4：公测与增长基础建设

**时间：** 2027-02-01 至 2027-06-30

### 阶段目标

- 从“工程完成”切到“产品开始运营”。
- 发布公开测试版本。
- 建立增长、留存、风控、监控的最低运营系统。

### 关键任务

1. 上线公测版：
   - 邀请码或限量注册
   - 版本公告
   - 反馈入口
2. 建立事件埋点和漏斗分析：
   - 注册完成率
   - 首帖完成率
   - 首次关注完成率
   - 7 日留存
3. 推出推荐与趋势第一版：
   - 热门内容
   - 推荐用户
   - 推荐话题
4. 完善运营能力：
   - 审核规则
   - 风控阈值
   - 封禁与申诉流程
   - 黑白名单机制
5. 完善性能与可用性：
   - 慢查询治理
   - 错误告警
   - 备份恢复演练
   - 静态资源策略
6. 文档与支持体系：
   - 用户帮助中心
   - FAQ
   - 管理台操作手册

### 交付物

- 公测版
- 运营埋点与增长面板
- 第一版推荐与趋势能力
- 风控、告警、备份流程

### 退出条件

- 项目具备面对真实外部用户的基本运营能力。
- 审核、申诉、封禁、回滚、告警流程可以跑通。
- 推荐与趋势至少达到“可用”而非“占位”。

### 阶段成功指标

- 注册用户数达到 1000。
- 日活跃用户达到 200 至 300。
- 7 日留存达到 25% 以上。
- 公测期严重线上故障可在 2 小时内恢复。

## 6.6 Phase 5：V1.0 正式版与移动端准备

**时间：** 2027-07-01 至 2027-10-31

### 阶段目标

- 发布 Web `V1.0` 正式版。
- 完成 `v2` 切换收尾。
- 做移动端入口，但仍以低风险方案优先。

### 关键任务

1. 完成 `v2` 切换收尾：
   - `v2` 覆盖率达到 95% 以上
   - `v1` 只保留极少数兼容层
   - 分批删除废弃 `v1` 路由
2. 发布 Web `V1.0`：
   - 稳定版本号管理
   - changelog
   - 正式版本说明
3. 推进移动端优先级顺序：
   - 优先 PWA
   - 再评估原生或跨端方案
4. 做移动端关键适配：
   - 登录
   - 首页流
   - 发布
   - 通知
   - 群组入口
5. 统一设计系统与交互规范：
   - 常用表单
   - 列表
   - 详情
   - 空状态
   - 错误状态
6. 建立正式发版节奏：
   - 月度小版本
   - 季度大版本

### 交付物

- Web `V1.0` 正式版
- 高覆盖率 `v2 API`
- `v1` 废弃计划与清理结果
- PWA 或移动适配版

### 退出条件

- `v1` 不再是日常开发主线。
- Web 正式版稳定发布。
- 移动端入口完成第一阶段验证。

### 阶段成功指标

- `v2` 覆盖率 >= 95%
- 线上错误率控制在 1% 以下
- Web 正式版具备连续 2 个发布周期稳定性

## 6.7 Phase 6：国际化、AI、商业化验证

**时间：** 2027-11-01 至 2028-04-30

### 阶段目标

- 在产品主线稳定后，推进长期价值建设。
- 以“小步验证”方式引入 AI 和商业化。
- 让国际化从界面翻译升级为产品能力。

### 关键任务

1. 国际化深化：
   - 不仅支持中英文界面
   - 梳理通知、邮件、错误信息、帮助中心的多语言体系
   - 建立翻译发布流程
2. AI 辅助功能：
   - AI 审核辅助建议
   - 话题与标签推荐
   - 搜索结果摘要
   - 创作者草稿辅助
3. 轻量商业化验证：
   - 高级主题
   - 个性化装扮
   - 群组增强权限
   - 创作者订阅或赞助实验
4. 生态与开放能力评估：
   - 开放 API 的必要性
   - Webhook 或集成能力的必要性
5. 合规与透明化：

## 9. 发布与验收机制

## 9.1 迭代节奏

- 每 2 周一个 Sprint
- 每 4 周一个阶段检查
- 每季度一个对外可演示版本

## 9.2 版本门槛

任一阶段对外发布前，至少满足：

- `lint` 通过
- `typecheck` 通过
- `build` 通过
- 关键 API 冒烟通过
- 高风险变更具备回滚方案
- 发布说明与已知问题清单已准备

## 9.3 阶段复盘必须回答的问题

1. 本阶段是否真的消灭了一个结构性问题，而不是只增加了功能数量？
2. 是否减少了模拟实现和重复逻辑？
3. 是否提高了对真实用户可见的稳定性？
4. 是否引入了新的长期维护负担？

## 10. 风险与应对

| 风险 | 说明 | 应对策略 |
|------|------|----------|
| 范围失控 | 新想法持续插队，主线被打断 | 每阶段只允许少量插单，其他进入待排期池 |
| 迁移期双版本混乱 | `v1/v2` 并存导致前后端和数据难以定位 | 强制日志打版本标记，按接口族迁移 |
| 演示代码长期残留 | 页面看起来完成，实际上不可用 | 每个模拟页建立“正式接入截止日期” |
| 数据迁移出错 | 新旧表并存期最容易出问题 | 迁移批次号、对账脚本、双写校验、回滚脚本 |
| 质量闸门缺失 | 每次改动都靠手测 | 从 Phase 1 起建立自动化冒烟与质量脚本 |
| 过早商业化 | 核心体验未稳就引入付费 | 商业化严格放到 Phase 6 之后 |

## 11. 未来 90 天的立即执行清单

## 12. 最终结论

NekoTribe 的长期路线图，不应该理解成“继续加更多功能”，而应该理解成下面这条主线：

**2026 年先完成架构和数据收口，2027 年完成真实运营级产品闭环，2027 年下半年到 2028 年再做国际化、AI 和商业验证。**

如果这个顺序不变，项目会越来越稳。

如果在 `v2`、数据迁移、质量体系完成前继续无序加功能，后面每一步都会更贵。

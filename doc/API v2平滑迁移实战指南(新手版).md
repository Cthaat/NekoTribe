# API v2 平滑迁移实战指南（新手版）

## 1. 这份文档解决什么问题

你现在的担忧很正常：

- 后端要上 v2，但前端里已经有大量 `/api/v1/...` 调用
- 不可能一次性把所有前后端 URL 都改对
- 你希望系统在迁移期间还能稳定跑，不影响用户
- 你还不想把项目搞成“改到一半只能硬着头皮继续”的状态

这份文档会带你用“并存 + 渐进切换 + 可观测 + 可回滚”的方式完成迁移。

一句话原则：

**先让 v1 和 v2 同时可用，再一批一批切流量，不做一刀切。**

---

## 2. 这份文档适合谁

这份文档是给下面这类情况准备的：

- 你不是只想看“高层方案”，而是想知道下一步到底改哪个文件
- 你项目里已经有不少真实业务，不可能推倒重来
- 你希望迁移过程尽量稳，最好出问题还能切回去
- 你希望文档能让未来的自己、或者后面接手的人也看得懂

如果你只想知道最终的 v2 接口长什么样，请看：

- `doc/NekoTribe-V2接口与Oracle重构总设计.md`

如果你想知道“怎么从当前项目一步步迁到那个目标”，就看本文。

---

## 3. 先认识你当前项目的真实情况

根据当前代码结构，关键点是：

1. 后端路由主要在 `server/api/v1/**`
2. 前端请求入口有两类：
   - `composables/useApi.ts`
   - `composables/useApiFetch.ts`
3. 前端很多页面和组件直接写死了 `/api/v1/...`
4. 鉴权中间件 `server/middleware/auth.ts` 当前只明确放行 `/api/v1/auth/`
5. 后端不只是路径问题，还存在：
   - 一批动作型接口
   - 一批响应结构不统一的接口
   - 一批数据库表语义不清的接口
6. 当前项目里不只是 API 路由要迁移，还有一部分数据模型也要迁移：
   - `sessions`
   - `settings`
   - `statements`
   - `follows / blocks`
   - `tweets -> posts`
   - `hashtags -> tags`

这意味着：

- 你不能直接“全局替换成 v2”
- 你也不能只做“URL 映射”就以为迁移完成
- 你应该先做兼容层，再做接口适配，再做数据层迁移

---

## 4. 这次迁移到底改什么

很多人会误以为 API v2 迁移只是把：

- `/api/v1/users/me`

改成：

- `/api/v2/users/me`

但你这个项目不是这么简单。

你这次迁移至少涉及 5 个维度：

1. 路径变化
   - 例：`/tweets/send-tweets` -> `/posts`
2. HTTP 方法变化
   - 例：`logout GET` -> `DELETE`
   - 例：`refresh GET` -> `PATCH`
3. 请求参数位置变化
   - 例：搜索条件从 body 改到 query
4. 响应结构变化
   - 例：`data.userData.userInfo` -> 扁平化 `data`
5. 数据模型变化
   - 例：`n_follows` 拆表
   - 例：`n_user_sessions` 改为新 session 模型
   - 例：本地 JSON store 迁入 Oracle

所以真正的迁移不是“路径替换”，而是下面这四层一起改：

- 前端请求入口层
- 后端路由兼容层
- 响应适配层
- 数据库模型迁移层

---

## 5. 迁移目标（你要达成的状态）

### 5.1 目标 1：v1 和 v2 同时在线

- 老页面继续访问 v1，不报错
- 新功能或已迁移页面可访问 v2

### 5.2 目标 2：前端改动集中化

- 尽量不在几十个页面里手改 URL
- 通过统一请求封装来决定走 v1 还是 v2

### 5.3 目标 3：请求和响应都能被适配

- 不只是 path 可切
- method、query、body、响应字段也可控

### 5.4 目标 4：可观测、可回滚

- 能知道哪些接口还在走 v1
- 能知道哪些请求是“v2 路由 + v1 兼容数据”
- 任何时候都能把某个接口快速切回 v1

### 5.5 目标 5：数据库迁移不把业务卡死

- 新表可以先建，不影响老逻辑
- 旧表在一段时间内还能继续服务
- 写接口在切换前必须有清晰回滚边界

---

## 6. 成功迁移的判断标准

满足下面这些，才算迁移真正成功：

1. 前端不再需要到处硬改版本号
2. 新接口默认从统一映射层走 v2
3. 线上可以统计 v1/v2 使用比例
4. 出问题可在 5 分钟内切回读取流量
5. 写接口切换前有明确的数据迁移与回滚方案
6. 连续 7~14 天稳定后，再讨论下线 v1

---

## 7. 总体迁移策略（六阶段）

### 阶段 0：盘点和冻结边界

先不要急着写 v2 路由，先做下面几件事：

1. 盘点当前所有 v1 接口
2. 标出哪些接口只是“读”
3. 标出哪些接口会“写数据库”
4. 标出哪些接口会涉及“新旧表切换”
5. 约定迁移期间不要继续随意改 v1 契约

这一阶段的目标不是写代码，而是避免边迁边漂。

### 阶段 A：打基础（不改业务行为）

先做这几件事：

1. 增加 v2 路由目录：`server/api/v2/**`
2. 鉴权中间件支持 v2 auth 放行
3. 前端增加“接口版本决策层”
4. 前端增加“请求适配层”
5. 前端增加“响应适配层”
6. 后端日志加上版本标记

这一阶段完成后，用户无感知，系统行为尽量不变。

### 阶段 B：迁移第一批低风险只读接口

建议顺序：

1. `system/config` 这类只读接口
2. `users/me`
3. `users/{id}`
4. `users/search`

不要第一刀就砍 auth。

### 阶段 C：迁移“旧表复用型”写接口

这类接口特点是：

- 路由变了
- method 可能变了
- 但底层暂时还写同一批旧表

例如：

- `comments`
- `bookmarks`
- 一部分 `notifications`

### 阶段 D：迁移“新旧表切换型”接口

这类接口才是高风险部分：

- `auth/session`
- `follow/block`
- `tweets -> posts`
- `hashtags -> tags`
- `settings/sessions/statements` 从文件迁数据库

### 阶段 E：扩大迁移面并灰度切流

- 以“接口族”为单位迁移
- 每迁完一族，观察日志和错误率
- 先灰度读，再灰度写

### 阶段 F：下线 v1

当以下条件满足时再考虑：

- v2 覆盖率 >= 95%
- 连续 7~14 天无严重回归
- 没有关键页面依赖 v1
- 没有新旧表双写任务还在运行

## 8. 开始前必须补上的基础设施

如果下面这些没准备好，不建议开始迁移。

### 8.1 API 版本配置

建议至少有：

- `NUXT_PUBLIC_DEFAULT_API_VERSION=v1`

作用：

- 决定“无版本路径”默认走 v1 还是 v2
- 后期灰度时，不需要到处改页面代码

### 8.2 统一请求入口

你当前项目至少有两个请求入口：

- `composables/useApi.ts`
- `composables/useApiFetch.ts`

必须统一共享同一个 helper，不要各写一套。

建议抽出：

- `composables/api/resolveApiPath.ts`
- `composables/api/adaptRequest.ts`
- `composables/api/adaptResponse.ts`

### 8.3 接口级别开关

仅有“默认 v1 / 默认 v2”还不够。

你需要支持：

- 全局默认版本
- 模块级版本
- 单接口强制版本

例如：

```ts
{
  '/users/me': 'v2',
  '/users/search': 'v2',
  '/tweets/list': 'v1',
  '/auth/login': 'v1'
}
```

### 8.4 日志与监控

最少要能观察到：

- 请求最终走的是 v1 还是 v2
- 请求原始 path 是什么
- 映射后的 path 是什么
- 是否用了请求适配器
- 是否用了响应适配器
- 接口耗时和错误率

### 8.5 迁移台账

没有台账，迁移一定会漏。

最少要有：

- 接口级台账
- 模块级进度看板
- 数据库迁移台账

## 9. 核心迁移思想：不是只有路径解析器

很多人迁 API 时只做这个：

- `resolveApiPath(path)`

这只解决了“请求发去哪”。

但你这个项目还要同时补上下面两层：

### 9.1 请求适配层

作用：

- 处理 v1 和 v2 之间的 method 差异
- 处理 body / query 的位置差异
- 处理字段名变化

典型例子：

- `logout`
  - v1: `GET /api/v1/auth/logout`
  - v2: `DELETE /api/v2/auth/tokens/current`
- `refresh`
  - v1: `GET /api/v1/auth/refresh`
  - v2: `PATCH /api/v2/auth/tokens/current`
- `tweets/search`
  - v1 代码结构混乱
  - v2 应该统一成 `GET /posts?q=...`

### 9.2 响应适配层

作用：

- 在前端尚未完全切到 v2 数据结构前，把 v2 响应临时适配成旧页面还能吃的格式

典型例子：

- v1:

```json
{
  "success": true,
  "message": "获取成功",
  "data": {
    "userData": {
      "userInfo": {
        "username": "demo"
      }
    }
  }
}
```

- v2:

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "username": "demo"
  },
  "meta": null
}
```

如果页面一时半会还没改完，就可以由响应适配层先兜一下。

### 9.3 后端兼容层

作用：

- v1 和 v2 路由可以共存
- 但核心逻辑不要长期双份维护

推荐做法：

1. 把核心 SQL 和业务流程抽到 service / repository
2. `v1` 路由只做旧契约适配
3. `v2` 路由只做新契约适配

---

## 10. 路径解析器、请求适配器、响应适配器分别负责什么

### 10.1 路径解析器

只负责：

- 输入 path
- 输出最终访问的 `/api/v1/...` 或 `/api/v2/...`

规则建议：

1. 如果 path 已经是 `/api/v1/...` 或 `/api/v2/...`，保持不变
2. 如果 path 是 `/users/me` 这种无版本路径：
   - 查询接口级映射
   - 没命中则走默认版本

### 10.2 请求适配器

负责：

- 改 method
- 改 query / body
- 改字段名

例如：

| 逻辑接口 | 旧协议 | 新协议 |
| --- | --- | --- |
| logout | `GET /auth/logout` | `DELETE /auth/tokens/current` |
| refresh | `GET /auth/refresh` | `PATCH /auth/tokens/current` |
| users update | `PUT /users/me` | `PATCH /users/me` |
| tweets search | body/query 混乱 | `GET /posts?q=` |

### 10.3 响应适配器

负责：

- 把 v2 扁平数据临时转成旧页面可接受的结构
- 或反过来，在新页面未全面切换前保持兼容

使用原则：

- 能不用适配器最好
- 但在平滑迁移阶段，适配器是“救命带”

### 10.4 三者的调用顺序

建议顺序：

1. 业务方传逻辑 path
2. 路径解析器决定版本
3. 请求适配器改请求
4. 发请求
5. 响应适配器改响应
6. 返回给页面

## 11. 具体落地步骤（按顺序做）

### 第 1 步：统一 API 版本配置

#### 为什么先做这个

你现在很多地方写的是 `/api/v1/...`，如果没有统一配置，后面会越来越难管。

#### 怎么做

先约定一个环境变量：

- `NUXT_PUBLIC_DEFAULT_API_VERSION=v1`

然后在请求封装中增加默认版本读取逻辑。

#### 新手注意

- 这里不是立刻把所有 URL 改成 v2
- 只是先让“版本是可配置的”

---

### 第 2 步：给请求封装加共享路径解析器

#### 目标

让调用方可以写：

- 旧写法：`/api/v1/users/me`
- 新写法：`/users/me`

#### 建议规则

1. 绝对版本路径保持不变
2. 无版本路径根据映射规则解析
3. 两个 composable 共用同一套规则

#### 特别提醒

不要在 `useApi.ts` 和 `useApiFetch.ts` 各写一份 resolver。

必须共享同一个 helper。

---

### 第 3 步：加入请求适配器

这是原文里最缺的一步。

#### 目标

当 path 被切到 v2 时，自动帮你修正：

- method
- body
- query
- 字段名

#### 适合用请求适配器的场景

1. 只改 URL
2. URL + method 一起改
3. URL + method + body/query 一起改

#### 不适合只靠请求适配器的场景

1. 涉及文件上传协议完全变化
2. 涉及多阶段业务流程变化
3. 涉及数据库新旧模型差异非常大

---

### 第 4 步：加入响应适配器

#### 为什么需要

因为很多 v2 的目标响应结构和 v1 不一样。

如果你一边迁后端、一边迁前端页面，适配器会帮你把风险拆小。

#### 什么时候应该加

- 页面暂时还依赖旧字段
- 你又想先切到 v2 路由

#### 什么时候应该删掉

- 当前接口对应的前端页面已经完全按 v2 DTO 渲染

---

### 第 5 步：先改鉴权中间件兼容 v2

文件：

- `server/middleware/auth.ts`

#### 当前风险

只放行了 `/api/v1/auth/`，当你新增 `/api/v2/auth/...` 时可能被错误拦截。

#### 正确做法

放行规则同时支持：

- `/api/v1/auth/`
- `/api/v2/auth/`

或者写成版本无关匹配。

#### 新手检查点

- 登录
- 注册
- 刷新 token
- 登出
- 忘记密码
- 修改邮箱验证码

不要只测登录成功这一个场景。

---

### 第 6 步：后端先复制外壳，不长期复制核心逻辑

#### 不建议

- 把 v1 文件整份复制一份当 v2，然后长期双份维护

#### 建议

1. 把 SQL、校验、组装响应的核心逻辑抽到 service / repository
2. `v1` 和 `v2` 路由只做“入参适配 + 出参适配”

这样做的好处：

- 修 bug 只改一处核心逻辑
- v1/v2 行为一致性更好

---

### 第 7 步：先迁只读接口，再迁写接口

推荐顺序：

1. `system`
2. `users`
3. `notifications` 只读列表部分
4. `account` 只读部分
5. `comments / bookmarks` 这类旧表复用写接口
6. `auth`
7. `follow / block`
8. `tweets -> posts`

#### 为什么 auth 不建议最先迁

因为 auth 的失败代价太高：

- 登录不了
- 刷新失败
- 大量 401
- session 混乱

新手更适合先从“只读、低副作用、易验证”的接口建立信心。

---

### 第 8 步：建立迁移看板和接口级台账

#### 模块级看板

| 接口族 | v1 状态 | v2 状态 | 前端切流比例 | 负责人 | 备注 |
| --- | --- | --- | --- | --- | --- |
| auth | 运行中 | 未上线 | 0% | 你 | 高风险，后置 |
| users | 运行中 | 部分上线 | 30% | 你 | 先迁只读 |
| posts | 运行中 | 未上线 | 0% | 你 | 结构变化大 |

#### 接口级台账模板

| 逻辑接口 | v1 路径 | v2 路径 | 方法变化 | 请求变化 | 响应变化 | 数据库依赖 | 开关名 | 当前状态 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 当前用户信息 | `/api/v1/users/me` | `/api/v2/users/me` | 无 | 无 | 深层嵌套 -> 扁平 | `n_users` | `api.users.me` | 待迁移 |

这张接口级台账非常重要，因为真正出问题时，大家查的是它，不是回忆。

---

## 12. 数据库迁移怎么做才算平滑

这是原文最缺的一块。

API 平滑迁移和数据库平滑迁移不是一回事，但你这个项目两者是绑定的。

### 12.1 先给接口分 4 类

#### A 类：只改 path，不改数据模型

例子：

- 某些纯读取接口

这类最好迁，通常只要：

- 加 v2 路由
- 做好 path 映射
- 校验响应格式

#### B 类：改 path + 改 method/字段，但仍复用旧表

例子：

- `users/me`
- 某些 `notifications`
- 某些 `comments`

这类可以先不动数据库，只迁接口契约。

#### C 类：接口变了，底层表也要换，但可以先建新表

例子：

- `n_user_sessions -> n_auth_sessions`
- `n_follows -> n_user_follows + n_user_blocks`
- `n_media -> n_media_assets + n_post_media`

这类要走：

1. 新表先建好
2. 老表继续服务
3. 回填旧数据到新表
4. 校验数据正确
5. 选择双写或灰度切写
6. 观察稳定后再切读

#### D 类：旧存储不是数据库，需要迁入数据库

例子：

- `userSettingsStore.ts`
- `sessionsStore.ts`
- `statementsStore.ts`

这类一定要补数据初始化和兼容读取逻辑。

### 12.2 推荐数据库迁移顺序

1. 先建新表，不切流量
2. 写一次性回填脚本
3. 做数据核对
4. 必要时做短期双写
5. 先切读，再切写，或者先切写再切读，必须按接口类型决定

对当前 V2 Oracle 基线，建议把下面几件事当成数据库迁移硬门槛：

1. 新表必须先补齐主键、唯一约束、外键和关键 `CHECK` 约束。
2. 高频外键列必须有前导索引，避免迁移期间联表查询、删除父记录或回填校验拖慢线上库。
3. `n_auth_sessions` 只能保存 `refresh_token_hash`、`access_jti`、`session_id`，不能回填 token 原文。
4. `n_media_assets.storage_key`、`n_group_invites.invite_code` 这类业务唯一键必须先去重再加唯一约束。
5. 回填后必须执行统计信息收集，避免优化器继续按旧数据分布生成执行计划。

### 12.3 什么是 shadow table

shadow table 就是：

- 新结构表已经建好
- 但暂时不让线上主流程直接依赖

这样好处是：

- 出问题时不影响当前业务
- 你可以先回填，再校验，再切流量

### 12.4 什么情况需要双写

只有当下面条件同时满足，才考虑双写：

1. 新旧表要并存一段时间
2. 旧逻辑暂时还不能下线
3. 这段时间里还会持续发生写入

比如：

- follow / block
- auth session

### 12.5 什么情况不要双写

不要把双写当默认选项。

以下情况更适合“停旧写 + 单写新表”：

- 旧存储本身只是本地 JSON
- 旧结构已经明显错误
- 双写成本很高且容易不一致

### 12.6 V2 数据库基线专项检查

执行或评审 `doc/neko_tribe-oracle-v2.sql` 时，至少检查：

| 检查项 | 通过标准 |
| --- | --- |
| 执行环境 | `ALTER SESSION SET CONTAINER`、tablespace 路径、用户密码已按目标环境确认 |
| 认证安全 | `n_auth_sessions`、`n_auth_otp_events` 不保存 token/验证码原文，也不授予通用只读账号 |
| 业务约束 | 会话 JTI、媒体 `storage_key`、群组 `slug`、邀请码等业务键有唯一约束 |
| 引用完整性 | 头像媒体、帖子父子关系、评论根评论、群组对象都有外键或明确的应用层约束说明 |
| 查询性能 | 列表过滤字段、外键列、时间排序列已有组合索引 |
| 读模型 | 公开资料和当前用户私密资料分开，公开资料不暴露 `last_login_at`，不能把 `v_user_profile_self` 当公开接口数据源 |
| 统计信息 | 所有 V2 表都纳入 `DBMS_STATS.GATHER_TABLE_STATS` |
| 物化视图 | 分析类物化视图默认按需刷新，生产刷新策略单独评审 |

## 13. 写接口的灰度与回滚边界

这是你迁移时必须最清楚的一件事。

### 13.1 读接口的回滚通常很简单

如果是纯读接口：

1. 映射层切回 v1
2. 保留 v2 代码不删
3. 查清问题后再修

### 13.2 写接口的回滚不是“切路由”这么简单

如果 v2 已经写入新表结构，仅仅把前端切回 v1，不代表系统就安全了。

你要先判断：

- 新写的数据有没有同步回旧表
- 旧读逻辑会不会读不到新写的数据
- 有没有生成不兼容的数据格式

### 13.3 回滚边界表

| 接口类型 | 能否直接切回 v1 | 注意事项 |
| --- | --- | --- |
| 纯读接口 | 可以 | 风险最低 |
| 写旧表的 v2 接口 | 通常可以 | 但要核对响应契约 |
| 写新表且旧表不感知 | 不可以直接切 | 先处理数据兼容 |
| 双写接口 | 可切，但要检查双写一致性 | 日志必须可追踪 |

### 13.4 推荐回滚触发条件

满足任一条件就要暂停或回滚：

1. 核心流程失败率显著上升
2. 出现数据写错或数据丢失风险
3. 登录态异常导致大量 401
4. 新旧表数据明显不一致
5. 某个页面在 SSR 或 CSR 任一路径出现大面积报错

## 14. 可观测方案（不做这个，迁移会盲飞）

### 14.1 最少要记录哪些字段

建议每个 API 请求都记录：

- `raw_path`
- `resolved_path`
- `api_version`
- `api_family`
- `adapter_request_used`
- `adapter_response_used`
- `method`
- `status_code`
- `duration_ms`
- `db_model_version`
- `feature_flag`
- `rollback_group`

### 14.2 推荐增加的 header

响应头可以增加：

- `x-api-version: v1 | v2`
- `x-api-family: users.me`
- `x-api-adapter: request | response | both | none`

这样排查问题非常快。

### 14.3 当前项目里可以落在哪

你当前已经有：

- `server/plugins/03-log.ts`
- `server/plugins/04-global-hooks.ts`

推荐：

- 在请求进入时记录原始 path
- 在封装层把“解析后的版本”放进请求头
- 在服务端日志里打印最终版本和耗时

### 14.4 如何统计 v2 覆盖率

可以按下面两种粒度看：

- 模块级覆盖率
  - 例如 `users`、`posts`
- 接口级覆盖率
  - 例如 `/users/me`

建议至少统计：

- 最近 24 小时 v1 调用次数
- 最近 24 小时 v2 调用次数
- 某接口 v2 占比
- v2 接口错误率、P95/P99 耗时
- 涉及新表接口的数据库慢查询数量
- 双写接口的新旧写入差异数量

数据库层建议额外记录迁移批次号，例如 `migration_batch_id`。回填脚本、双写日志、校验 SQL 和回滚脚本都使用同一个批次号，排查时不会混在一起。

## 15. 可直接执行的最小方案（推荐你先落这个）

如果你只想先安全起步，按下面做：

1. `server/api/v2/` 先创建目录
2. `server/middleware/auth.ts` 支持 v2 auth 放行
3. 抽出共享 `resolveApiPath(path)`
4. 抽出共享 `adaptRequest(config)`
5. 预留 `adaptResponse(result)`
6. 先迁一个最简单接口，比如 `users/me`
7. 前端仅把这个接口改成“无版本路径调用”
8. 记录日志，看它最终走的是 v1 还是 v2

只要这个闭环跑通，后面基本就是重复这个模板。

## 16. 一个完整样板：以 `/users/me` 为例

这一节非常适合你拿来照着做。

### 16.1 为什么选它

因为它有这些优点：

- 读取类接口
- 业务链路短
- 容易测试
- 风险相对低

### 16.2 当前状态

- v1 路由：`/api/v1/users/me`
- 旧响应结构有深层嵌套
- 前端可能直接使用旧路径

### 16.3 迁移目标

- 新增 v2 路由：`/api/v2/users/me`
- 前端逻辑调用改成无版本路径：`/users/me`
- 默认先让它继续走 v1
- 当 v2 准备好后，只切这一条接口的映射

### 16.4 建议迁移步骤

#### 第一步：新增 v2 路由外壳

- 创建 `server/api/v2/users/me.get.ts`

#### 第二步：抽公共逻辑

把获取用户信息的 SQL 和映射逻辑提出来，不让 v1/v2 各自维护一套。

#### 第三步：定义 v2 DTO

目标结构例如：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "user_id": 1001,
    "username": "demo",
    "display_name": "Demo",
    "avatar_url": "/upload/avatars/1001/avatar.png"
  },
  "meta": null
}
```

#### 第四步：前端把调用改成逻辑 path

不要继续写死：

```ts
'/api/v1/users/me'
```

而是写成：

```ts
'/users/me'
```

#### 第五步：先让映射仍指向 v1

这样页面不变，但入口已经收口。

#### 第六步：切这一个接口到 v2

把映射改成：

- `/users/me` -> `v2`

#### 第七步：验证

至少测：

1. 已登录获取成功
2. 未登录返回 401
3. 页面渲染不报错
4. 日志里能看到已走 v2

### 16.5 如果页面还依赖旧结构怎么办

先用响应适配器做临时兼容。

目标不是永远保留适配器，而是：

- 让迁移可拆步骤
- 让前后端不必同一天改完

### 16.6 回滚方式

如果出问题：

1. 把 `/users/me` 的映射切回 v1
2. 保留 v2 路由不删
3. 修复后再切回来

## 18. 测试清单（每次迁移都要过一遍）

### 18.1 功能正确性

1. 同一业务场景，v1 与 v2 返回是否满足预期
2. 错误码是否一致，尤其：
   - `401`
   - `403`
   - `404`
   - `422`
   - `500`
3. 分页字段、时间字段、空值字段是否一致

### 18.2 鉴权与会话

1. 登录后访问 v2 受保护接口是否通过
2. token 刷新流程是否仍工作
3. 未登录访问是否正确返回 401
4. Cookie 模式和 Bearer 模式是否都正常
5. 多设备、多标签页是否会话异常

### 18.3 前端兼容

1. 页面是否仍能渲染
2. 是否有字段改名导致的 undefined
3. 是否有旧页面还依赖深层嵌套结构

### 18.4 Nuxt 项目专项检查

1. SSR 首屏是否正常
2. CSR 页面切换是否正常
3. `useApi.ts` 和 `useApiFetch.ts` 是否一致走 resolver
4. 同一个接口在服务端和客户端调用时结果是否一致

### 18.5 文件上传专项检查

1. 头像上传是否正常
2. 媒体上传是否正常
3. 删除媒体是否误删物理文件
4. 数据库记录与文件路径是否一致

### 18.6 性能与稳定性

1. 同接口 v1/v2 的耗时对比
2. 错误日志是否显著增加
3. 数据库慢查询是否增加
4. 是否出现 Oracle 连接池异常增长

### 18.7 数据一致性检查

针对迁移中的新旧表，至少要确认：

1. 回填数量一致
2. 新写数据是否进入预期表
3. 双写时是否存在漏写
4. 回滚后旧逻辑能否读到必要数据
5. 新表唯一约束是否有历史脏数据冲突
6. 只读账号是否无法读取认证会话、验证码、当前用户私密资料视图
7. 回填后统计信息是否已收集，关键接口是否没有新增全表扫描

### 18.8 Oracle 基线执行检查

1. 开发环境可完整执行 V2 基线脚本，或已记录哪些初始化段需要按环境注释。
2. 所有 `CREATE TABLE` 对象都有对应统计信息收集项。
3. 所有新增唯一约束都有数据去重方案。
4. 物化视图创建前的基础数据和统计表已经准备好。
5. `neko_readonly` 授权清单经过安全复核。

## 19. 常见坑（新手最容易踩）

1. 只改后端路径，忘了改鉴权放行规则
2. 只做路径解析，不做 method/body/response 适配
3. 只测成功路径，不测失败路径
4. v2 返回字段改名后，前端没做兼容
5. 两个请求入口各写一套 resolver，后面互相打架
6. 一次迁太多接口，定位问题困难
7. 写接口已经切到新表，却以为“切回 v1 path 就算回滚”
8. 没有日志标签，出了问题不知道请求到底走了哪一层
9. 新旧表并存时没有台账，后面根本不知道哪些还在双写

## 21. 验收标准（做到这些才算迁移成功）

### 单接口完成标准

1. 已有 v2 路由
2. 已纳入映射层控制
3. 已完成请求适配
4. 如有必要，已完成响应适配
5. 有日志可确认最终版本
6. 已通过测试清单
7. 已写入接口级台账

### 单模块完成标准

1. 模块内所有接口已盘点
2. 高风险接口已明确数据库迁移路径
3. 灰度期间错误率稳定
4. 可以一键切回模块级 v1
5. 数据库约束、索引、统计信息和只读授权已通过专项检查

### 全站完成标准

1. 前端不再到处硬编码版本号
2. 新接口默认走 v2
3. v2 覆盖率持续稳定
4. 没有关键页面依赖 v1
5. 没有遗留双写逻辑
6. 连续 7~14 天稳定
7. V2 数据库基线、设计文档和迁移台账保持一致

## 22. 给你的最后建议

你现在不是“技术不够”，而是“迁移策略必须分层”。

只要记住这几条，你就不会乱：

1. **先并存，再切流，不要一刀切**
2. **先改入口层，不要先改全量页面**
3. **路径解析器不够，还要有请求适配层和响应适配层**
4. **读接口先迁，写接口后迁**
5. **写接口切换前，先明确数据库回滚边界**
6. **每次只迁一小块，并且可观测、可回滚**

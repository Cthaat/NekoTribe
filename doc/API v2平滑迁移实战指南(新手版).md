# API v2 平滑迁移实战指南（新手版）

## 这份文档解决什么问题

你现在的担忧很正常：

- 后端要上 v2，但前端里已经有大量 `/api/v1/...` 调用
- 不可能一次性把所有前后端 URL 都改对
- 你希望系统在迁移期间还能稳定跑，不影响用户

这份文档会带你用“并存 + 渐进切换 + 随时回滚”的方式完成迁移。

一句话原则：**先让 v1 和 v2 同时可用，再一批一批切流量，不做一刀切。**

---

## 先认识你当前项目的真实情况

根据当前代码结构，关键点是：

1. 后端路由主要在 `server/api/v1/**`
2. 前端请求入口有两类：
   - `composables/useApi.ts`（`apiFetch`）
   - `composables/useApiFetch.ts`（`useApiFetch`）
3. 前端很多页面和组件直接写死了 `/api/v1/...`
4. 鉴权中间件 `server/middleware/auth.ts` 当前只明确放行 `/api/v1/auth/`

这意味着：

- 你不能直接“全局替换成 v2”
- 你应该先做“兼容层”，然后逐步迁移

---

## 迁移目标（你要达成的状态）

### 目标 1：v1 和 v2 同时在线

- 老页面继续访问 v1，不报错
- 新功能或已迁移页面可访问 v2

### 目标 2：前端改动集中化

- 尽量不在几十个页面里手改 URL
- 通过统一请求封装来决定走 v1 还是 v2

### 目标 3：可观测、可回滚

- 能知道哪些接口还在走 v1
- 任何时候都能把某个接口快速切回 v1

---

## 总体迁移策略（四阶段）

## 阶段 A：打基础（不改业务行为）

先做这三件事：

1. 增加 v2 路由目录：`server/api/v2/**`
2. 鉴权中间件支持 v2 auth 放行
3. 前端增加“接口版本决策层”（只在请求封装改）

这一阶段完成后，用户无感知，系统行为不变。

## 阶段 B：迁移第一批低风险接口

建议顺序：

1. `system/config` 这类读接口
2. `users/me` 这类低副作用接口
3. 再到 tweets、notifications 等复杂接口

每次只迁一小组，迁完立刻验证。

## 阶段 C：扩大迁移面

- 以“接口族”为单位迁移（auth、users、tweets、follow、notifications）
- 每迁完一族，观察日志和错误率

## 阶段 D：下线 v1

当以下条件满足时再考虑下线：

- v2 覆盖率 >= 95%
- 连续 7~14 天无严重回归
- 没有关键页面依赖 v1

---

## 具体落地步骤（按顺序做）

## 第 1 步：统一“API 版本配置”

### 为什么要先做这个

你现在很多地方写的是 `/api/v1/...`，如果没有统一配置，后面会越来越难管。

### 怎么做

先约定一个环境变量（例如）：

- `NUXT_PUBLIC_DEFAULT_API_VERSION=v1`

然后在请求封装中增加默认版本读取逻辑（在 `useApi.ts`、`useApiFetch.ts` 中使用）。

### 新手注意

- 这里不是立刻把所有 URL 改成 v2
- 只是先让“版本是可配置的”

---

## 第 2 步：给请求封装加“路径解析器”

### 目标

让调用方可以写：

- 旧写法：`/api/v1/users/me`（继续可用）
- 新写法：`/users/me`（自动补版本）

### 建议规则

1. 如果 path 已经是 `/api/v1/...` 或 `/api/v2/...`，保持不变
2. 如果 path 是 `/auth/login`、`/users/me` 这种无版本路径：
   - 按映射规则决定走 v1 还是 v2
3. 映射规则建议支持“接口级别开关”

例如：

- `/users/me` -> v2
- `/tweets/list` -> v1

这样你只需改映射，不用全局改页面代码。

---

## 第 3 步：先改鉴权中间件兼容 v2

文件：`server/middleware/auth.ts`

### 当前风险

只放行了 `/api/v1/auth/`，当你新增 `/api/v2/auth/login` 时可能被错误拦截。

### 正确做法

放行规则同时支持：

- `/api/v1/auth/`
- `/api/v2/auth/`

或写成版本无关的匹配（如 `/api/v{n}/auth/` 概念）。

### 新手检查点

- 登录、注册、刷新 token、登出都要分别测试
- 不要只测登录成功这一个场景

---

## 第 4 步：后端先复制“外壳”，核心逻辑复用

### 不建议

- 把 v1 代码整份复制到 v2 然后长期双份维护

### 建议

1. 把 SQL、校验、组装响应的核心逻辑抽到 `server/utils` 或 service 文件
2. `v1` 和 `v2` 路由只做“入参适配 + 出参适配”

这样做的好处：

- 修 bug 只改一处核心逻辑
- v1/v2 行为一致性更好

---

## 第 5 步：每次只迁移一个“接口族”

推荐顺序：

1. auth
2. users
3. account
4. tweets
5. interactions
6. notifications
7. follow

每个接口族都按这 5 个动作执行：

1. 新建 v2 路由
2. 对齐输入输出契约
3. 在请求映射层把该接口切到 v2
4. 手测 + 回归
5. 观察日志 24 小时

---

## 第 6 步：建立迁移看板（非常重要）

建议在文档里维护一个表格：

| 接口族 | v1 状态 | v2 状态  | 前端切流比例 | 负责人 | 备注     |
| ------ | ------- | -------- | ------------ | ------ | -------- |
| auth   | 运行中  | 已上线   | 100%         | 你     | 稳定     |
| users  | 运行中  | 部分上线 | 30%          | 你     | 继续迁移 |
| tweets | 运行中  | 未上线   | 0%           | 你     | 待开始   |

这能避免“我以为改完了，其实还有漏网接口”。

---

## 可直接执行的最小方案（推荐你先落这个）

如果你只想先安全起步，按下面做：

1. `server/api/v2/` 先创建目录
2. `server/middleware/auth.ts` 支持 v2 auth 放行
3. 在 `useApi.ts` 和 `useApiFetch.ts` 增加统一 `resolveApiPath(path)`
4. 先迁一个最简单接口（如 config 或 users/me）
5. 前端仅把这个接口改成“无版本路径调用”验证机制可用

只要这个闭环跑通，你后面就是重复这个模板，不会失控。

---

## 测试清单（每次迁移都要过一遍）

## 功能正确性

1. 同一业务场景，v1 与 v2 返回是否满足预期
2. 错误码是否一致（尤其 401/403/422/500）
3. 分页字段、时间字段、空值字段是否一致

## 鉴权与会话

1. 登录后访问 v2 受保护接口是否通过
2. token 刷新流程是否仍工作
3. 未登录访问是否正确返回 401

## 回归风险

1. 首页、用户页、发帖、评论、通知、个人设置全部走一遍
2. 检查是否有因字段名变化导致的前端渲染报错

## 性能与稳定性

1. 同接口 v1/v2 的耗时对比
2. 错误日志是否显著增加
3. 数据库慢查询是否增加

---

## 回滚方案（你必须提前准备）

迁移不是“永不失败”，所以要先设计回滚。

## 快速回滚方式

1. 在请求映射层把某接口从 v2 切回 v1
2. 保留 v1 路由，短期内不要删除
3. 回滚后记录问题原因，再修复 v2

## 回滚触发条件（建议）

满足任一条件立即回滚：

1. 核心流程失败率明显上升
2. 出现数据写错或数据丢失风险
3. 登录态异常导致大量 401

---

## 常见坑（新手最容易踩）

1. 只改后端路径，忘了改鉴权放行规则
2. 只测成功路径，不测失败路径
3. v2 返回字段改名后，前端没做兼容
4. 并发场景没测，导致偶发错误
5. 一次迁太多接口，定位问题困难

---

## 你可以照抄的迁移节奏（每周）

## 第 1 周

- 完成基础设施：版本配置、路径解析器、auth 放行兼容
- 迁移 1~2 个低风险读接口

## 第 2 周

- 迁移 users/account 相关接口
- 增加错误监控和覆盖率统计

## 第 3~4 周

- 迁移 tweets/interactions/notifications
- 每周做一次回归总检

## 第 5 周

- 评估 v1 剩余流量
- 准备 v1 下线计划

---

## 验收标准（做到这些才算迁移成功）

1. 前端不再需要到处硬改版本号
2. 新接口默认从映射层走 v2
3. 线上可观测到 v2 覆盖率稳步提升
4. 出问题可在 5 分钟内切回 v1
5. 连续 7~14 天稳定后再讨论下线 v1

---

## 给你的最后建议

你现在不是“技术不够”，而是“迁移策略需要分层”。

只要记住这三点，你就不会乱：

1. **先并存，再切流，不要一刀切**
2. **先改入口层，不要先改全量页面**
3. **每次只迁一小块，并且可回滚**
